## 1. ゴールとKPI

### ユーザー体験フロー（2段階CV構造）

```
【広告クリック】
     ↓
①LP上のフォーム送信（名前・メール）━━━━━━━━━━━━▶ ★主CV（オンライン計測）
     ↓
②サンクス画面表示
     ↓
③自動返信メール（TimeRex予約リンク）
     ↓
④TimeRexで日時選択 ━━━━━━━━━━━━━━━━━━━━━━▶ ★最終CV（オフライン計測）
     ↓
⑤カレンダー自動登録・リマインド送信
     ↓
⑥説明会参加・成約（スプレッドシート管理）
```

外部の予約ツール単体だと「予約しなかった人の連絡先」が失われるため、**フォーム→自動返信の2段階**設計で全員の連絡先を確保しつつ日程調整を自動化している。

### CVとKPIの定義

| 項目 | 内容 |
|------|------|
| **★主CV（Google広告の最適化対象）** | `step_form_complete`（GA4）— LP内でオンライン計測 |
| **★最終CV（事業KPI）** | TimeRex予約完了 — **スプレッドシート手動管理** |
| **副CV（マイクロCV）** | `cta_click`（GA4）※学習材料に |
| **主要KPI** | CPA（申込1件あたりの広告費）/ CPO（最終成約1件あたりの広告費） |
| **副次KPI** | CTR / スクロール到達率 / フォーム開始率 / **フォーム完了→TimeRex予約完了の歩留まり** |

**CPA目安の考え方**:
- 個人向け価格7万円 × 成約率 = LTV想定
- 例: 申込→参加率60% × 参加→成約率30% = 申込→成約率18%
- 申込1件あたり許容CPA = 成約LTV × 目標利益率 ÷ 申込→成約率
- 実数値はデータ蓄積後に再計算（Ads-Phase Dで見直し）

### 計測方針（重要）

- **Google広告の自動入札**は `step_form_complete`（オンラインCV）で最適化する
- **事業の本当の成果**（TimeRex予約完了・説明会参加・成約）は、[remaining-tasks 4-1](remaining-tasks.md) の**スプレッドシート手動管理**で把握する
- **オフラインコンバージョンインポート**（Google広告にGCLID+CVをCSVアップロードする仕組み）は**月CV50件超 + 広告費月10万円超**を条件に、Ads-Phase D で導入判断する
- ただし **GCLID保持ロジックは最初から実装しておく**（取得だけしてスプレッドシートに貯めれば、将来オフラインCV導入時にすぐ使える）

## 5. UTM / GCLID 設計

> [remaining-tasks 3-4](remaining-tasks.md) と兼ねる。広告の流入識別・オフラインCV連携に必須。

### 命名規則（UTM）

```
utm_source   = google           （広告媒体: google / yahoo / facebook 等）
utm_medium   = cpc              （cpc=有料検索 / display / social 等）
utm_campaign = 20260414_search_unexperienced  （YYYYMMDD_種別_テーマ）
utm_content  = ad_A / ad_B      （広告バリエーション識別）
utm_term     = {keyword}        （Google広告の動的挿入でキーワード自動付与）
```

### GCLID保持（将来のオフラインCV導入に備えた先行実装）

Google広告クリック時にURLに自動付与される識別子（例: `?gclid=Cj0KCQiA...xyz`）。これをフォーム送信時に一緒に保存しておくと、後日TimeRex予約完了と突合して広告側に「このクリックは最終的に予約まで到達した」と報告できる。

**重要**: GCLIDは**今すぐ使わなくても、取得だけは最初からしておく**。後から遡って取得することは不可能なため。

### 自動返信メール内TimeRexリンクのUTM

自動返信メール内のTimeRexリンクには以下のUTMを付与する：

```
https://timerex.net/s/xxxxx?utm_source=email_autoreply&utm_medium=email&utm_campaign=form_followup
```

これによりGA4で「フォーム送信→TimeRex到達」の歩留まりが計測可能になる（TimeRex側の計測設定は別途必要）。

### フォーム→Web3Formsへの引き渡し項目

LP到達時にUTMとGCLIDを `sessionStorage` に保存し、フォーム送信時にhidden fieldで送信：

| 項目 | 用途 |
|------|------|
| `utm_source` / `utm_medium` / `utm_campaign` / `utm_content` / `utm_term` | 流入元識別 |
| `gclid` | Google広告クリック識別子（オフラインCV用） |
| `landing_page` | 最初に到達したURL（リマケ分析用） |
| `referrer` | 直前のURL |

これらを [remaining-tasks 4-1](remaining-tasks.md) のスプレッドシートに蓄積することで、将来のオフラインCV連携（Ads-Phase D）がスムーズに。

### 実装タスク

| # | 作業 | 担当 | 状態 |
|---|------|:----:|:----:|
| U-1 | [script.js](../script.js) にUTM・GCLID取得・localStorage保存（90日）ロジック追加 | 🔧 | ✅ 2026-04-14 |
| U-2 | [index.html](../index.html) のフォームにUTM・GCLID用hidden input追加（8項目） | 🔧 | ✅ 2026-04-14 |
| U-3 | Web3Forms側で全項目がメール本文に含まれるか確認 | 🤝 | ⬜ Web3Formsキー設定後 |
| U-4 | 自動返信メール文面のTimeRexリンクにUTM付与 | 👤 | ⬜ TimeRex URL発行後 |
| U-5 | スプレッドシート運用ルール整備（フォーム送信ログとTimeRex予約の突合） | 👤 | ⬜ |

### U-1 / U-2 実装メモ（2026-04-14）

- **保存先**: `localStorage`（キー: `tb_attribution`）
- **保存期間**: 90日（Google推奨のGCLID最大保持期間に合わせた）
- **上書きロジック**: URLに新しい `utm_*` または `gclid` があれば既存データを上書き（最新の広告クリックを優先）
- **取得項目**: `utm_source` / `utm_medium` / `utm_campaign` / `utm_content` / `utm_term` / `gclid` / `landing_page` / `referrer`
- **送信経路**: `<form>` 内のhidden input → `FormData` に自動収容 → Web3Forms経由でメール送信
- **プライベートモード等でlocalStorage無効時**: エラーを握りつぶして通常動作を継続（try/catchで防御）

## 6. 予算・入札戦略

### 初期予算（配信開始〜データ蓄積期）

**採用方針: 月5万円スタート（日予算 約¥1,600）**

| 想定 | 日予算 | 月予算 | 狙い |
|------|--------|--------|------|

| **★採用** | **¥1,600〜¥1,700** | **¥50,000** | **月15〜25クリック想定。まず動かしてキーワード反応を見る** |

| 加速 | ¥5,000〜 | ¥150,000〜 | 早期にtCPA移行可能なCV量を確保 |

### 月5万円スタートでの想定と留意点

| 項目 | 想定値 | 備考 |
|------|--------|------|
| 想定CPC | ¥200〜¥400 | 「エンジニア 未経験 転職」等は競合多く¥500超もあり得る |

**重要**: 月5万円予算では自動入札の学習が進みにくいため、**キーワード精度と広告文の質**で勝負する方針になります。広く打つより**意図の明確な少数キーワード**に予算を集中させることが成功のカギです。

### 入札戦略の段階移行

```
配信開始          ━━━━━━━━━━━━━━━━━▶ クリック数の最大化（手動CPC上限付き）

CV 月30件以上     ━━━━━━━━━━━━━━━━━▶ 目標コンバージョン単価（tCPA）
CV 月50件以上     ━━━━━━━━━━━━━━━━━▶ tROAS or P-MAX拡張検討
```

**注意**: 入札戦略を変更すると2週間程度の学習期間が発生し、一時的にCPAが悪化することがあります。頻繁に戻さず、最低2週間は様子を見る。

|------|------|:----:|
| **毎日（初期2週間）** | 配信量・CV発火・LP表示エラーの有無 | 🤝 |
| **週次** | 検索語句レポート、除外キーワード追加、CTR・CV・CPA推移、**フォーム送信→TimeRex予約の歩留まり確認**（スプレッドシート突合） | 🤝 |
| **月次** | キャンペーン全体の費用対効果、広告文アセットの入替、予算配分見直し、**CPO（予約完了単価）算出** | 🤝 |
| **四半期** | 入札戦略の見直し、キャンペーン拡張/削減、LPの大幅改修判断 | 🤝 |

## 8. リスク・注意点

| リスク | 対策 |
|--------|------|
| **計測未設定のまま配信** | Ads-Phase A完了前は配信禁止。P-1〜P-5の前提条件を必ず満たす |
| **広告文とLPの不一致で直帰率悪化** | 広告見出しと[FV](../index.html)コピーの整合を毎月チェック |
| **除外キーワード不足で無駄打ち** | 週次の検索語句レポート確認を運用ルールに組み込む |
| **入札戦略の過度な変更** | 変更後2週間は学習期間として静観 |
| **クリック詐欺・競合クリック** | Google広告の自動無効クリック検出を信頼。異常値だけ手動で確認 |
| **LP側のフォーム不具合** | 配信前に実機でフォーム送信テスト。Web3Formsのメール到達も確認 |
| **TimeRex予約への歩留まり悪化** | フォーム送信数だけ増えてもTimeRex予約率が低いと成果に繋がらない。自動返信メール文面・TimeRex到達率をスプレッドシートで週次チェック |
| **GCLIDの取りこぼし** | UTM対応漏れ・SPAのような遷移でパラメータが消えるとオフラインCV導入時に困る。配信前に実クリックテストで`sessionStorage`への保存を確認 |
